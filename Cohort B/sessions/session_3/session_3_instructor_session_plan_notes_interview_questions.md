## 10–20 min: Architecture Breakdown — What Are We Adding and Why

### Instructor Goal

Before generating any code, students must understand the full auth flow on a whiteboard or shared diagram.

### Draw or Show This Flow

```
Client sends POST /auth/login
  ↓
FastAPI reads email + password from request body
  ↓
Query User table for matching email
  ↓
passlib verifies plain password against bcrypt hash
  ↓
python-jose creates signed JWT (sub=user_id, role=user.role, exp=now+30min)
  ↓
Server returns {"access_token": "eyJ...", "token_type": "bearer"}

---

Client sends GET /tickets with Authorization: Bearer eyJ...
  ↓
get_current_user dependency extracts Bearer token from header
  ↓
python-jose decodes and verifies signature using SECRET_KEY
  ↓
Token claims are read: sub, role, exp
  ↓
If expired or invalid signature → 401 Unauthorized
  ↓
If valid → route handler receives current_user object
  ↓
Route filters tickets by current_user.id (or all tickets if admin)
```

### Key Architecture Points to Explain

1. JWT is stateless — the server does not store sessions; the token carries all needed claims.
2. The signature uses HMAC-SHA256 with a server-side SECRET_KEY — without the key, you cannot forge a valid token.
3. FastAPI's Depends() system is the correct pattern for auth — do not put auth logic inside the route handler.
4. Hashing is one-way — you cannot reverse a bcrypt hash; you can only verify a plain password against it.
5. Role is embedded in the JWT — the server reads it from the token, not from the DB on every request.

### Ask Before Moving On

- What is the difference between authentication and authorization?
- When does the token get created? When does it get verified?
- What happens if someone tampers with the payload of a JWT?

## 95–105 min: Concept Pause — Authentication vs Authorization, JWT, Password Hashing, Protected Routes

### Instructor Goal

Consolidate implementation into interview-level understanding.

### Explain the Core Concepts in Technical Plain Language

Authentication is the process of verifying identity — "who are you?" The client proves identity by presenting credentials (email + password). The server issues a token after successful verification.

Authorization is the process of verifying permissions — "what are you allowed to do?" The server checks the authenticated user's role or claims before executing a sensitive operation.

JWT (JSON Web Token) has three Base64url-encoded sections separated by dots: header (algorithm), payload (claims: sub, role, exp, iat), and signature (HMAC of header+payload using SECRET_KEY). The server verifies the signature on every request. If the signature is valid, the payload has not been tampered with.

bcrypt is a one-way hashing function designed to be computationally expensive. It includes a random salt per hash, so the same password hashes differently each time. `passlib.CryptContext` handles salt generation and verification automatically.

FastAPI's `Depends()` system enables middleware-like behavior at the route level. `get_current_user` is a dependency function that extracts the token, decodes it, queries the DB, and returns the User object. If it raises an HTTPException, the route handler never executes.

### Student Writing Task

What is the difference between authentication and authorization in your API?

Authentication is verifying that the requester is who they claim to be — in our API, this happens at POST /auth/login when we verify the password and issue a JWT. Authorization is verifying that the authenticated user has permission to perform an action — in our API, this happens in the require_admin dependency that checks `current_user.role == "admin"` before allowing PATCH /tickets/{id}.

## Common Student Mistakes

These are the exact errors students will hit during the session:

1. `422 Unprocessable Entity on POST /auth/login` — Student is sending `{"email": ..., "password": ...}` as JSON. If the route uses `OAuth2PasswordRequestForm`, it expects `application/x-www-form-urlencoded`. Fix: either use a Pydantic schema for JSON login (simpler for this project) or tell students to use form data in Swagger.

2. `jose.exceptions.JWTError: Signature verification failed` — Student changed the SECRET_KEY after generating tokens. Fix: regenerate a token with the current key.

3. `AttributeError: 'NoneType' object has no attribute 'id'` on ticket creation — get_current_user is returning None instead of raising HTTPException. Fix: ensure the user lookup block raises `credentials_exception` when `session.get(User, user_id)` returns None.

4. `sqlalchemy.exc.OperationalError: no such column: user.hashed_password` — Student added the User model but did not drop and recreate the SQLite database (or add a migration). Fix: delete `tickets.db`, restart the server, tables will recreate.

5. `ImportError: cannot import name 'get_current_user' from 'app.auth.dependencies'` — Circular import between `database.py` and `auth/dependencies.py`. Fix: import `get_db` inside the function body, or restructure the import chain.

6. `bcrypt: (trapped) error reading bcrypt version` warning — Harmless but confusing. This is a known passlib/bcrypt version mismatch warning. Fix: `pip install bcrypt==4.0.1` or ignore.

7. `GET /tickets returns all tickets for a regular user` — Student forgot to add the filter `where(Ticket.owner_id == current_user.id)` for non-admin users. Fix: add conditional filter in the route.

8. `PATCH /tickets/{id} returns 200 for a regular user` — require_admin dependency was defined but not added to the route signature. Fix: add `_: User = Depends(require_admin)` to the handler parameters.

9. `KeyError: 'sub' when decoding JWT` — Token was created without the `sub` field in the payload dict. Fix: confirm `create_access_token({"sub": str(user.id), "role": user.role})`.

10. `passlib.exc.UnknownHashError: hash could not be identified` — Student accidentally stored the plain password instead of the hash. Fix: verify that `hash_password(user.password)` is called before writing to the DB in the signup route.

# Questions to Discuss: Interview Perspective

## Basic Project Questions

### Q1. What authentication mechanism did you implement?

Expected answer:

We implemented JWT-based authentication using the `python-jose` library. On login, the server issues a signed JWT containing the user's id, role, and expiration time. On every subsequent request to a protected endpoint, the client sends the token in the `Authorization: Bearer` header. The server decodes and verifies the signature using a secret key. If the token is valid and not expired, the request proceeds. This is a stateless mechanism — the server does not maintain any session state.

### Q2. How do you store passwords securely in your system?

Expected answer:

We use `passlib` with the bcrypt algorithm to hash passwords before storing them in the database. bcrypt is intentionally slow and computationally expensive, which makes brute-force attacks costly. It also generates a random salt for each hash, so two users with the same password will have different hashes stored in the database. On login, `CryptContext.verify(plain_password, stored_hash)` is called — we never reverse the hash or store the plain password anywhere.

### Q3. What is the difference between authentication and authorization?

Expected answer:

Authentication answers "who are you?" — it is the process of verifying the identity of the requester. In our system, this happens at `POST /auth/login` where we verify the email and password and issue a JWT. Authorization answers "what are you allowed to do?" — it is the process of checking whether the authenticated user has permission to perform a specific action. In our system, this happens in the `require_admin` dependency that checks `current_user.role == "admin"` before allowing access to `PATCH /tickets/{id}`. These are two separate concerns and they are implemented in separate dependency functions.

### Q4. What does the get_current_user dependency do?

Expected answer:

`get_current_user` is a FastAPI dependency function that runs before the route handler on every protected endpoint. It extracts the Bearer token from the `Authorization` header using `OAuth2PasswordBearer`. It then calls `decode_access_token()` which uses `python-jose` to verify the JWT signature and decode the payload. If the signature is invalid or the token is expired, it raises an `HTTPException` with status 401. If the token is valid, it extracts the `sub` claim (user_id), queries the database for that user, and returns the User object to the route handler. This gives every protected route handler access to the authenticated user via `current_user: User = Depends(get_current_user)`.

### Q5. How does role-based access control work in your ticket API?

Expected answer:

We have two roles: "user" and "admin". The role is stored in the User table and embedded in the JWT payload at login. We have a `require_admin` dependency function that calls `get_current_user` and then checks whether `current_user.role == "admin"`. If not, it raises `HTTPException(status_code=403, detail="Admin access required")`. This dependency is injected into `PATCH /tickets/{id}`. For `GET /tickets`, there is no separate dependency — we check the role inside the route handler and apply a filter: regular users get only their own tickets, admins get all tickets.

## Technical Deep-Dive Questions

### Q6. What is inside a JWT token and how does the server verify it?

Expected answer:

A JWT has three parts separated by dots: header, payload, and signature. The header is a Base64url-encoded JSON object specifying the algorithm, typically `{"alg": "HS256", "typ": "JWT"}`. The payload is a Base64url-encoded JSON object containing claims — in our implementation these are `sub` (user_id), `role`, `exp` (expiration timestamp), and `iat` (issued at). The signature is `HMAC-SHA256(base64url(header) + "." + base64url(payload), SECRET_KEY)`. The server verifies the token by recomputing the signature using its SECRET_KEY and comparing it to the signature in the token. If they match, the payload has not been tampered with. `python-jose` handles this automatically in `jwt.decode()`.

### Q7. Why does your login endpoint return 401 for both "email not found" and "wrong password"?

Expected answer:

Returning different error messages for these two cases — for example, "Email not found" vs "Incorrect password" — allows an attacker to enumerate valid user accounts. If they get "Email not found" for `attacker@test.com` but "Incorrect password" for `victim@company.com`, they now know the second email is a registered account and can target it with a brute-force attack. By returning the same message — "Incorrect email or password" — for both cases, we prevent account enumeration. This is a standard security practice. The same principle applies to signup: do not reveal whether a given email is registered to an unauthenticated caller — instead, just return "Email already registered."

### Q8. What happens to your API if the SECRET_KEY is leaked?

Expected answer:

If the SECRET_KEY is leaked, an attacker can forge valid JWT tokens with arbitrary claims — including `{"sub": "1", "role": "admin"}` — that will pass signature verification on our server. This would give them full admin access to every endpoint. The correct response is to immediately rotate the SECRET_KEY, which invalidates all currently issued tokens. Since JWT is stateless and the server does not maintain a token store, there is no way to revoke individual tokens without additional infrastructure like a token blocklist in Redis. This is one of the core trade-offs of stateless JWT: you cannot invalidate a token before it expires. In production, this is mitigated by using short expiration times and refresh token rotation — but we are not implementing refresh tokens in this session.

### Q9. How does FastAPI's Depends() system work for auth, and why is it better than putting auth logic inside the route handler?

Expected answer:

FastAPI's `Depends()` is a dependency injection system. When you declare `current_user: User = Depends(get_current_user)` in a route function signature, FastAPI calls `get_current_user` before executing the route handler and injects its return value. If `get_current_user` raises an `HTTPException`, the route handler never runs. This is better than putting auth logic inside handlers for several reasons: it is reusable across multiple routes without code duplication; it is composable — `require_admin` can call `Depends(get_current_user)` internally; it is testable in isolation; and it separates concerns cleanly. The route handler focuses only on business logic and can assume `current_user` is a valid authenticated user.

### Q10. Walk me through the exact sequence of a POST /tickets request from an authenticated user.

Expected answer:

The client sends `POST /tickets` with the `Authorization: Bearer <token>` header and a JSON body `{"title": "...", "description": "..."}`. FastAPI's router matches the path and method. Before calling the route handler, it resolves dependencies — `get_current_user` is called. `OAuth2PasswordBearer` extracts the token from the header. `jwt.decode()` verifies the signature and checks expiration. The `sub` claim is extracted and used to query `SELECT * FROM user WHERE id = <sub>`. The resulting `User` object is injected as `current_user`. Inside the route handler, a new `Ticket` is created with `owner_id=current_user.id` and written to the database via the SQLModel session. A 201 response is returned with the created ticket. If the token was missing, the response would have been 401 before the handler ever ran.

## System Design and Trade-off Questions

### Q11. JWT is stateless. What are the trade-offs compared to server-side sessions?

Expected answer:

JWT is stateless because the server does not need to store any session data — the token itself carries all needed information and the server only needs the SECRET_KEY to verify it. This scales horizontally without shared session storage, which is a major operational advantage. The trade-off is that individual tokens cannot be revoked before they expire. If a user logs out or their account is compromised, there is no built-in way to invalidate the token unless you maintain a blocklist (typically in Redis), which reintroduces state. Server-side sessions (stored in a database or Redis) allow instant revocation but require shared storage across server instances. For our support ticket copilot with short-lived tokens, stateless JWT is the correct choice.

### Q12. Why do we embed the role in the JWT payload instead of looking it up from the database on every request?

Expected answer:

Embedding the role in the JWT avoids a database query on every authenticated request. If we stored only the user_id in the token and fetched the role from the DB on every request, we would add latency and DB load proportional to request volume. The trade-off is that if a user's role is changed in the database after their token is issued, the token still carries the old role until it expires. For a support ticket system where role changes are infrequent and tokens are short-lived (30 minutes), this stale-role window is an acceptable trade-off. For systems requiring instant role revocation, you either use very short-lived tokens, maintain a token revocation list, or fetch the role from DB on each request.

### Q13. What is the difference between 401 Unauthorized and 403 Forbidden? When does each appear in your API?

Expected answer:

401 Unauthorized means the request lacks valid authentication credentials. Despite its name, it is about authentication, not authorization. In our API, a 401 is returned when: no `Authorization` header is present, the token signature is invalid, or the token is expired. The response includes `WWW-Authenticate: Bearer` to tell the client how to authenticate. 403 Forbidden means the request is authenticated but the server refuses to authorize the action. In our API, a 403 is returned when a regular user attempts `PATCH /tickets/{id}`, which is restricted to admins. The distinction matters operationally: a 401 tells the client to obtain credentials; a 403 tells the client that even with valid credentials, this action is not permitted for their role.

### Q14. If you had to scale this auth system to 100,000 users, what would you change?

Expected answer:

Several changes would be needed at scale. First, we would move the SECRET_KEY and any configuration to a proper secrets management service (AWS Secrets Manager or HashiCorp Vault) rather than an environment variable on each server. Second, we would add refresh tokens with a database-backed token store so users are not constantly re-entering credentials when short-lived access tokens expire. Third, we would add a Redis-backed token blocklist to support instant revocation on logout or account compromise. Fourth, we would consider moving password hashing to a background task (since bcrypt is intentionally slow) to avoid blocking the login endpoint. Fifth, we would add rate limiting on `POST /auth/login` to prevent brute-force attacks — libraries like `slowapi` integrate with FastAPI easily.

### Q15. Why do we use bcrypt instead of SHA-256 or MD5 for password hashing?

Expected answer:

SHA-256 and MD5 are general-purpose cryptographic hash functions designed to be fast. This is a disadvantage for password hashing: an attacker with a leaked database and a GPU can compute billions of SHA-256 hashes per second, making brute-force or rainbow table attacks feasible. bcrypt is specifically designed for password hashing and is intentionally slow — its cost factor can be increased as hardware gets faster. It also incorporates a random salt per password, which eliminates rainbow table attacks. passlib's bcrypt implementation handles salt generation automatically and its `verify()` function performs a constant-time comparison to prevent timing attacks. MD5 is completely broken for any security use. SHA-256 without salt is vulnerable to rainbow tables. bcrypt (or argon2, which is the current recommended alternative) is the correct choice for password storage.
